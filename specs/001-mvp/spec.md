# crypto-riddle — MVP 仕様書（spec.md 初版）

## 0. ゲートP（プロトタイプ）結果の記録

- **判定: 続行**（2026-07-21 代表）。使い捨てプロトタイプ（`prototype/gate-p` ブランチ、第1話「アルファテック社 顧客データ流出事件」）で、導入→探索(カード取得)→解決(暗号解読／攻撃特定／防衛)のコアループを検証。
- **代表フィードバック（本 spec に反映）:**
  1. 基本方針は良い。
  2. **サポート役キャラを増やす**（探索の案内・解決の伴走）。
  3. **解決シナリオと物語を組み合わせる**（パズルと事件ストーリーを一体化する）。
- プロトタイプで見えた弱点（本 spec で対処）: ①暗号が1種類のみで「組み合わせ感」が弱い ②判定が厳密一致で硬い ③誤答ペナルティ無しで緊張感が薄い。

## 1. 概要

情報処理安全確保支援士（SC）レベルのセキュリティ知識が、遊びながら身につく**謎解き×防衛アドベンチャー**。
プレイヤーはセキュリティ担当者として、**1マップ＝1インシデント事件**を「導入 → 探索 → 解決」の3パートで解き明かし、
攻撃手段を特定して防衛策を講じることでマップを攻略していく（全10〜20マップ想定）。
核となる手触りは「集めたヒント（カード）と暗号を**パズルのように組み合わせて**攻撃を読み解く」推理体験。

- プラットフォーム: **Web アプリ（スマホブラウザ対応）**（concept 決定を踏襲）。
- 正式名称: **仮称 crypto-riddle**。確定はゲート①判断事項（§11）。

## 2. 背景・目的

- 座学で定着しにくいセキュリティ知識を、事件解決の達成感と反復で継続学習に変える。
- 代表は 2026年11月に **ネットワークスペシャリスト（NW）試験**を受験予定。セキュリティ分野は NW でも配点が大きく、本ゲームの学習が試験対策に直結する（study_agent・NW学習計画と連携）。
- 将来的に SC 試験対策そのもの、六創堂の教育・研修コンテンツへの転用も視野。

## 3. ユーザーストーリー

- **US-1（学習者・スキマ時間）**: スマホで1事件を10〜15分で解き、SCレベルの攻撃手法を「解けた」実感とともに覚えたい。
- **US-2（学習者・弱点把握）**: 解いた事件の分野（暗号/認証/Web/インシデント対応 等）が記録され、弱点分野が分かるようにしたい。
- **US-3（学習者・物語で継続）**: 登場人物やサポート役との会話・事件の続きが気になって次のマップに進みたい。
- **US-4（監修者＝代表）**: IPA 過去問（SC 午前II・午後 直近5年）を素材に、出典を明示しつつ事件シナリオを量産・監修したい。

## 4. コアゲームループ（GDD の核）

```
事件発生（導入）
  → 現場を探索して人物/ログ/機器から手がかりを得る（ヒントカード獲得）
  → 得たカードと暗号を組み合わせて「攻撃手段」を特定（推理）
  → 正しい防衛策を選んでインシデントを解決（防衛）
  → 事件クリア＝物語が一歩進む／XP・分野習熟が加算
  → 次の事件（次マップ）へ
```

### 4.1 1マップの3パート構成（逆転裁判・ダンガンロンパ型）

| パート | 目的 | 具体 |
|--------|------|------|
| **導入** | 事件の前提提示 | 背景・登場人物・被害会社の状況をストーリーで提示。**サポート役キャラ**が事件概要と目的を説明 |
| **探索** | 手がかり収集 | 現場（人物・PC・ログ・物証）を調べ **ヒントをカード形式で獲得**。ダミーカードを混在させ取捨選択を作る |
| **解決** | 推理と防衛 | (a) 暗号解読ミニゲーム → (b) カード組合せで攻撃手段特定 → (c) 防衛策選択。サポート役が誤答時にフォロー |

## 5. 登場人物・サポート役（フィードバック反映：新設）

物語とパズルを一体化させ、探索・解決に「人」を絡める。

- **主人公（プレイヤー）**: 新任のセキュリティ担当。プレイヤーの分身。
- **サポート役（相棒キャラ）**: 逆転裁判の助手枠。役割 = ①導入で事件と目的を説明 ②探索で「ここを調べてみては」の**ヒント誘導** ③解決で誤答時に**軽いフォローと考え方の提示**（答えは言わない） ④クリア後に**知識の要点を解説**（学習定着）。
  - 学習ゲームとしての「教える機能」をキャラ台詞に載せることで、無機質な正誤判定を物語体験に変える。
- **事件関係者（マップ固有）**: 被害企業の社員・経営者・容疑者等。証言がヒントカード／引っ掛けになる。関係や動機が探索で明らかになると同時に攻撃手段が絞られていく（＝物語とパズルの一体化）。

> サポート役のキャラ造形（1体か複数か、性格・口調）と、関係者を各マップ何人置くかは §11 で判断。

## 6. シナリオ構造（フィードバック反映：物語×パズルの一体化）

- **1マップ = 1事件 = 1つの物語アーク**。導入で提示された「謎（誰が・どうやって・なぜ攻撃したか）」が、探索での人間関係・動機の解明と、暗号/技術的手がかりの解明の**両輪**で解けていく。
- 技術パズル（暗号・攻撃特定）を物語から切り離さない：カードは必ず「誰の証言／どの機器のログ」という**物語上の出所**を持ち、正解は「事件の真相」と一致する。
- **マップ間の連続性（任意）**: 事件をまたぐ横糸（黒幕・組織など）を薄く通し、次マップへの牽引にする（MVP では必須にしない。§11 判断）。

## 7. 探索・ヒントカード設計

- 各マップに **調査ポイント複数**（人物・PC・ログ・物証）。クリックで調査 → カード獲得。
- **カード種別（案）**: 証言 / ログ / 通信記録 / ニュース・外部情報 / 暗号文 / 鍵・手がかり / 対策カード（解決用）。
- **ダミーカード**を混在させ「もっともらしいが今回の直接原因ではない」引っ掛けを作る（SCで問われる“表面的に怪しい選択肢に飛びつかない”思考の訓練）。
- カード枚数・正解に必要な枚数・ダミー比率は §11 でバランス判断（プロトタイプは全10箇所・正解6/ダミー4）。

## 8. 解決パート設計（本作の核 = 暗号をパズルのように組み合わせる）

### 8.1 暗号解読ミニゲーム
- 手がかりから暗号文と鍵/ヒントを得て解読（プロトタイプはシーザー暗号のシフトをライブ調整）。
- **弱点対処①**: 1マップ内で**暗号を複数種・複数段組み合わせる**方向を検討（例: 古典暗号で得た文字列を鍵に別処理、ハッシュ照合、XOR、Base64 等）。暗号方式の配分（古典/現代）は §11。

### 8.2 攻撃手段の特定
- 集めたカードから正解の組合せを選んで攻撃手段を断定。
- **弱点対処②**: 判定を「厳密一致のみ」から緩和するか（部分点・複数正解ルート・段階ヒント）は §11 判断（concept の「一意解か複数解か」未決事項）。

### 8.3 防衛策の選択
- 攻撃手段に対する有効な対策カードを選択（本質的でない対策を誤答肢に）。クリアで事件解決。

### 8.4 難易度・緊張感
- **弱点対処③**: 誤答への軽いペナルティ（試行回数・信頼度ゲージ・タイマー等）を入れるか、学習ゲームとして無制限試行を許容するかは §11 判断。

## 9. ゲーム内経済・進行・クリア条件

- **報酬**: 事件クリアで XP。ダミー回避・ノーヒントクリア等にボーナス（案）。
- **進行**: マップを順に解放。分野別（暗号/認証/Web/攻撃手法/インシデント対応/法制度）に習熟度を記録し弱点を可視化。
- **クリア条件（マップ）**: 攻撃手段の特定＋有効な防衛策の選択に成功。
- **学習指標**: 分野別習熟、SC 合格ライン（正答率60%）対応のランク表示（concept 由来。MVP採否は §11）。
- **スコープ整理**: ドリル（午前II型）・間隔反復（復習モード）は **初期スコープ外**（リリース後追加候補）。MVP は謎解きマップ攻略に集中。

## 10. 機能要件 / 非機能要件 / 受け入れ基準 / スコープ外

### 機能要件（MVP）
- FR-1 1マップの3パート（導入/探索/解決）を通しでプレイできる。
- FR-2 探索で調査ポイントからカードを獲得し、インベントリで確認できる。
- FR-3 暗号解読ミニゲームを解ける（最低1方式、拡張余地を持つ設計）。
- FR-4 カード組合せで攻撃手段を特定し、防衛策を選んでクリアできる。
- FR-5 サポート役キャラが導入説明・探索誘導・誤答フォロー・クリア解説を行う。
- FR-6 事件クリアで XP・分野習熟が加算・保存される。
- FR-7 IPA 過去問由来素材に**出典表記**を各所で明示する。
- FR-8 MVP マップ本数（初期公開分）を用意する（本数は §11）。

### 非機能要件
- NFR-1 スマホブラウザでスキマ時間に遊べる軽さ。
- NFR-2 学習履歴の永続化（端末ローカル → 将来アカウント同期）。
- NFR-3 IPA 過去問の利用は出典明記を必須とし、改変時の表記規則を定める（§11）。

### 受け入れ基準
- [ ] 未整理の§11判断事項が代表レビューで解決される。
- [ ] MVP スコープ（謎解きマップ攻略型・サポート役・物語一体化）が承認される。
- [ ] 承認をもって status を review→approved にし、次工程 plan.md（ゲート②）へ。

### スコープ外（当面）
- 実攻撃ツール・実環境演習（設問内で完結）。
- 対人対戦。
- ドリル/間隔反復（復習モード）。

## 11. ゲート①で判断する事項（要判断リスト）

代表レビューで一括判断する。

1. **正式名称の確定**（現在 crypto-riddle は仮称）。
2. **サポート役キャラ**: 1体か複数か／性格・口調／どこまで喋らせるか（学習解説の量）。
3. **物語の横糸**: マップ間に黒幕・連続ストーリーを通すか（MVP必須か任意か）。
4. **暗号パズルの深さ**: 1マップで暗号を複数種・複数段組み合わせるか。古典/現代の配分。
5. **解決の判定方式**: 厳密一致のままか、部分点・複数正解ルート・段階ヒントを許すか。
6. **誤答ペナルティ**: 試行制限/信頼度ゲージ/タイマーを入れるか、無制限試行を許容するか。
7. **カードバランス**: 1マップの調査ポイント数・正解/ダミー比率の目安。
8. **1マップのプレイ時間目安**（レベル・分量・カード枚数の物差し）。
9. **MVP マップ本数**と、シナリオ制作体制（AI生成＋代表監修で量産か、代表原案か）。
10. **IPA過去問の出典表記の具体形式**（改変利用時を含む）。
11. **合格ライン対応ランク等の学習指標**を MVP に入れるか。
12. **ゲームエンジン利用の検討**（代表要望）— ※技術判断のため plan.md（ゲート②）で候補比較＋advisor相談のうえ決定。ここでは「エンジン導入を選択肢に含める」方針のみ確認。
13. **terra-town との開発優先順位**。
